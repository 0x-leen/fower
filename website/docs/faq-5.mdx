## 什么是原子属性 (Atomic Props)

原子类是简单的单一用途的样式单位。与内联样式很像，原子样式仅应用单个样式声明。与内联样式不同，原子样式具有较低的特异性，使其易于覆盖，并且可以通过使用伪类，媒体查询等进行修改。

## 为什么使用 CSS in JS

作为老前端，我也曾经长时间沉迷于“关注点分离”。但在过去的几年，组件化成为时代主流，CSS-in-JS 逐渐在 React 社区兴起并流行。CSS In JS 有一些传统 CSS 没有的特性：

- 天生的 Scoped 样式

  有效地大规模构建 CSS 是非常困难的，这已不是秘密。当加入一个现有的长期存在的项目时，通常会发现 CSS 是系统中最难发现的部分。
  为了解决这个问题，在 CSS 社区投巨资试图解决这些问题，使我们的样式类似的方法更容易维护 OOCSS 由妮可沙利文和 SMACSS 由乔纳森·斯努克-但明显的赢家，现在的人来说似乎是 BEM，或来自 Yandex 的块元素修饰符。
  最终，BEM（纯粹应用于 CSS）只是一种命名约定，它选择将样式限制为遵循.Block\_\_element--modifier 模式的类。在任何给定的 BEM 样式的代码库中，开发人员都必须记住始终遵循 BEM 的规则。如果严格遵循，BEM 确实可以很好地发挥作用，但是为什么像范围界定这样的根本性问题留给纯约定俗成呢？

  而 CSS in JS 是天生的 Scoped 样式。

- 支持非浏览器，如 React Native, 小程序

  我们团队是 RN 的深度用户

- 编程性

  JS 可以完美代替 Less/Sass/Stylus

- 组件化

- 更好的开发体验

  你不需要写 CSS，只需写 JS，心智负担更小，而且可以配合 TypeScript，类型提示更加精准。

在组件体系结构中，将多个组件的关注点放在一个地方就成为了当务之急。

## Server-side Rendering ?

For years, developers have been fighting with complexity and related bugs of cascading style sheets. I have seen many styles, written in dozens various methodologies or without, worse or better, but almost always hard to manage and scale without rigorous supervision. Despite Less, SASS, and the other CSS preprocessors aimed to help, something was still broken in the land of styles.

多年以来，开发人员一直在与级联样式表的复杂性和相关的错误作斗争。我见过许多样式，用数十种不同的方法编写，有无，有无，或更差或更好，但几乎总是很难在没有严格监督的情况下进行管理和扩展。尽管 Less，SASS 和其他 CSS 预处理程序旨在提供帮助，但样式方面仍然存在一些问题。

死码
由于 HTML 和 CSS 之间存在隐式关系，因此通常很难跟踪未使用的 CSS 规则并通知作者或将其从捆绑中删除。通常，您不知道规则在哪里使用。例如，可能是多个代码库，也可能有条件地从用于生成 HTML 的语言中应用了类名，或者类名已由客户端 JavaScript 操纵。
随着时间的推移，无效代码可能会对站点性能和开发人员理解代码的能力产生负面影响。
借助 JavaScript 中的显式，可跟踪变量和模块，我们可以在顶部实现解决方案，从而在 CSS 规则和 HTML 元素之间创建显式连接。
CSS-in-JS 有助于删除无效代码。

CSS-in-JS 使开发人员更具表现力，同时鼓励使用比级联更可维护的模式。

基于状态的样式
CSS-in-JS 支持的非常强大的模式之一是基于状态的样式。从技术上讲，通常将其实现为 JavaScript 函数，该函数接收状态对象并返回 CSS 属性。结果，生成了与元素状态相对应的 CSS 规则。与更传统的方式（其中我们构建包含多个类名的类属性）相比，它具有一些优点： 1.负责最终 CSS 规则的逻辑可以访问状态，并且可以与其他样式一起放置。 2.生成 HTML 的逻辑被类级联逻辑所取代。
这两点都应该使复杂的与状态相关的 CSS 更具可读性。
CSS-in-JS 使开发人员 API 可以比使用一堆条件类名称更好的方式描述基于状态的样式。
