## 为什么使用 CSS in JS

作为老前端，我也曾经长时间沉迷于“关注点分离”。但在过去的几年，组件化成为时代主流，CSS-in-JS 逐渐在 React 社区兴起并流行。CSS In JS 有一些传统 CSS 没有的特性：

- 天生的 Scoped 样式

  有效地大规模构建 CSS 是非常困难的，这已不是秘密。当加入一个现有的长期存在的项目时，通常会发现 CSS 是系统中最难发现的部分。
  为了解决这个问题，在 CSS 社区投巨资试图解决这些问题，使我们的样式类似的方法更容易维护 OOCSS 由妮可沙利文和 SMACSS 由乔纳森·斯努克-但明显的赢家，现在的人来说似乎是 BEM，或来自 Yandex 的块元素修饰符。

  最终，BEM（纯粹应用于 CSS）只是一种命名约定，它选择将样式限制为遵循.Block\_\_element--modifier 模式的类。在任何给定的 BEM 样式的代码库中，开发人员都必须记住始终遵循 BEM 的规则。如果严格遵循，BEM 确实可以很好地发挥作用，但是为什么像范围界定这样的根本性问题留给纯约定俗成呢？

  而 CSS in JS 是天生的 Scoped 样式。

- 支持非浏览器，如 React Native, 小程序

  我们团队是 RN 的深度用户，使用 Styli 可以统一 UI 开发的体验。

- 编程性

  JS 可以完美代替 Less/Sass/Stylus，可以使用变量、条件、循环、函数等所有编程特性。

- 基于状态的样式

  CSS-in-JS 支持的非常强大的模式之一是基于状态的样式。从技术上讲，通常将其实现为 JavaScript 函数，该函数接收状态对象并返回 CSS 属性。结果，生成了与元素状态相对应的 CSS 规则。与更传统的方式（其中我们构建包含多个类名的类属性）相比，它具有一些优点： 1.负责最终 CSS 规则的逻辑可以访问状态，并且可以与其他样式一起放置。 2.生成 HTML 的逻辑被类级联逻辑所取代。
  这两点都应该使复杂的与状态相关的 CSS 更具可读性。
  CSS-in-JS 使开发人员 API 可以比使用一堆条件类名称更好的方式描述基于状态的样式。

- 组件化

  曾经 Web 应用是基于页面 (Page)的，一个页面的实现通常由 HTML、JS、CSS 3 个文件组成，那时关注点分离是合理的。现在，前端进入了组件化时代，一个页面通常由多个组件组合完成，这时 CSS in JS 更适合组件化的场景。

- 更好的开发体验

  你不需要写 CSS，只需写 JS，心智负担更小，而且可以配合 TypeScript，类型提示更加精准。 CSS-in-JS 使开发人员更具表现力，同时鼓励使用比级联更可维护的模式。在组件体系结构中，将多个组件的关注点放在一个地方就成为了当务之急。

- 强约束性

  多年以来，前端开发人员一直在与 CSS 的各种方法论作斗争，比如 BEM/SMACSS/OOCSS，但在没有严格监督的情况下都很难落实和管理展。Styli 提供一种提约束的编写样式方法: Atomic Props，让每个人使用同样的方式编写样式。

- 消除死代码

  由于 HTML 和 CSS 一般通过类名建立联系, 他们之间存在隐式关系，因此通常很难跟踪未使用的 CSS 样式，删除组件后，很难精确删除对应的 CSS，因为你会黑怕其他地方使用了样式，你会没信心删除，而且缺乏安全感。 使用 CSS-in-JS， 样式和结构的关系是显式的，几乎不会出现死代码。
  CSS-in-JS 有助于删除无效代码。
